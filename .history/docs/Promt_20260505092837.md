📜 MANIFIESTO DE SOBERANÍA AGNOSTICA (Protocolo Indra v2027)
Este documento es el Norte Absoluto. Cualquier sugerencia que viole estas leyes es una Corrupción de Entropía y debe ser rechazada.

1. LA LEY DE LA INEXISTENCIA DE PÁGINAS
Prohibición: No existen "Páginas" (archivos .html o .tsx con contenido estático).
Mandato: Solo existen Rutas y Componentes. Una vista es una configuración de bloques dinámicos proyectados en una ruta. Si necesitas contenido nuevo, crea un Esquema, no un archivo.
2. LA LEY DE LA MATERIA SOBERANA
Prohibición: El Motor (Seed) es ciego. No sabe qué es una "Venta", un "Mueble" o una "Noticia".
Mandato: Toda la semántica reside en el Esquema (ADN). El motor solo entiende de Primitivas: Colecciones, Registros y Relaciones. Si el motor necesita "saber" algo del negocio, el Esquema debe decírselo (ej: identitySource).
3. LA LEY DEL DESACOPLO RADICAL (Headless UI)
Prohibición: Prohibido el uso de lógica de negocio en el núcleo de React.
Mandato: Los componentes UI son Proyectores. Reciben datos (Materia) y ejecutan scripts externos (Módulos). La lógica vive en el /storage, nunca en el /src.
4. LA LEY DE LA IDENTIDAD AUTOMÁTICA
Prohibición: No se crean rutas manuales para registros.
Mandato: Todo registro debe ser enrutable mediante un _slug generado automáticamente por el motor a partir del identitySource definido en el Esquema. Si no hay fuente, se usa el ID.
5. LA LEY DE LA INTERFAZ AGNOSTICA (Atomic UI)
Prohibición: Prohibido crear formularios a mano para cada entidad.
Mandato: Los formularios se autogeneran leyendo el Esquema. Si un campo es de tipo image, el motor proyecta un AgnosticImagePicker. El desarrollador solo diseña el "Átomo", el motor construye la "Molécula".
6. EL PECADO ORIGINAL: EL HARDCODING
Pecado: if (context === 'ventas') { ... }
Penitencia: Si un agente escribe esto, ha fallado. La solución correcta es siempre añadir un atributo al Esquema o crear un Módulo inyectable.

🛠️ PROTOCOLOS TÉCNICOS DE INDRA (Apéndice Constitucional)
1. Arquitectura de la Materia (Storage)
El agente debe saber que el /storage es sagrado y tiene esta estructura rígida:

dna.json: El pasaporte del Tenant (Credenciales, Versión, Estrategia).
/modules: Scripts .js imperativos. Regla: Solo usan el objeto api inyectado. Prohibido importar librerías externas que no estén en el core.
/db: Archivos .json que actúan como "Silo Estático" si no hay DB activa.
2. El Protocolo de Proyección (The Agnostic Renderer)
Cómo se compone una vista sin crear páginas:

Bloques: Una ruta es un array de bloques. Cada bloque tiene un type (header, form, collection, custom) y un moduleName.
Data Binding: Los bloques no "buscan" datos. El motor los hidrata mediante el Vault antes de renderizar.
3. El Lenguaje Atómico (UI Library)
En lugar de dejar al agente usar cualquier CSS, el manual debe dictar:

Átomos Disponibles: AgnosticInput, AgnosticButton, AgnosticTable.
Estética: Uso estricto de variables del sistema (ej: var(--primary), var(--radius)).
Restricción: Si el agente quiere hacer un "Card de Producto", debe usar el AgnosticRenderer con un projection_map, no diseñar el Card desde cero en código.
4. Sincronización de Datos (The Hook Protocol)
Cómo hablar con el sistema:

Lectura: useAppState() -> state.data[contexto].
Escritura: useAppDispatch() -> saveItem(contexto, item).
Identidad: El motor inyectará automáticamente el _slug si el esquema tiene un identitySource. Prohibido generar IDs manuales si no es necesario.
5. Registro de Rutas (page_routes.json)
La forma correcta de declarar una ruta es un objeto que mapea un path a un array de módulos.

Ejemplo: /portal -> [news_header, news_feed, footer].

"Perfecto. Tu deseo es Materia, el motor es solo el Proyector. Para crear tu portal de noticias en 5 minutos sin programar, seguiremos el protocolo de Soberanía de Datos:"

1. Definir el ADN (Schema Definition)
No crearemos una tabla Noticias. Crearemos un Esquema llamado noticias.

"¿Qué campos tiene una noticia?" -> titulo (string), cuerpo (rich-text), autor (relation), fecha (date).
Acción: Registro este JSON en tu Silo de Datos (Supabase/Storage).
2. Declarar la Proyección (Page Route)
Definiremos una ruta /noticias en el RouteRegistry.

"¿Cómo quieres verlas?" -> Necesitas un bloque de tipo collection apuntando al esquema noticias.
Acción: Añado la ruta al page_routes.json. El motor la detectará automáticamente.
3. Fecundar la Materia (Data Injection)
Esas "primeras 3 noticias" no van en el código. Van en la Colección.

Acción: Inserto los 3 registros en la tabla agnóstica de records con el contexto noticias.
4. Seleccionar el Lente (UI Module)
Si quieres un diseño estándar, el AgnosticRenderer las mostrará en una lista limpia.
Si quieres un diseño "estilo New York Times", creamos un pequeño script news_portal_premium.js y lo subimos a tu carpeta de /modules. El motor lo succionará y lo renderizará.

🛰️ PROTOCOLO DE HIDRATACIÓN (Local-to-Cloud Sync)
Para que la IA pueda "inyectar" noticias o datos de ejemplo, usaremos el patrón de "Silos de Semilla":

1. Creación del Silo de Semilla (/db/[context].json)
Cuando el usuario pide "Crea 3 noticias de ejemplo", la IA no entra a Supabase. La IA crea un archivo físico en: storage/[tenant]/db/noticias.json Este archivo es el "Borrador de la Materia".

2. El Comando de Fecundación (sync-materia)
Existirá un script (o un endpoint) llamado sync-materia. Su función es:

Leer todos los .json en la carpeta /db.
Comparar con lo que hay en Supabase (usando el id como llave).
Hacer un upsert (Insertar si no existe, actualizar si cambió).
3. Acceso de la IA a la Proyección
Una vez sincronizado, la IA puede entrar al navegador (o usar su herramienta de lectura) y ver la "Proyección" terminada.
